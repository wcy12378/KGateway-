## 1. 目标

在不影响当前功能的前提下，给项目补充两条长期治理规则：

1. 单个代码文件超过 300 行时必须提醒并进入拆分评估。
2. 单个代码文件超过 500 行时必须拆分，除非有明确豁免。
3. 每个代码文件必须有文件级说明，讲清楚这个文件负责什么、不负责什么、依赖边界是什么。

这份 spec 只定义规则、实施步骤和验收标准。是否开始执行，由你最终拍板。

## 3. 文件行数治理规则

### 3.1 分级标准

| 行数 | 规则 | 动作 |
| ---: | --- | --- |
| `< 300` | 正常 | 不强制处理 |
| `300-499` | 黄色预警 | 必须在 review / spec 中提醒，并给出是否拆分的判断 |
| `>= 500` | 红色强制 | 必须拆分，除非写明豁免理由 |

### 3.2 计算范围

计入行数：

- 源码行
- 类型定义
- 注释
- 空行
- JSX / TSX 模板

不计入治理对象：

- `node_modules`
- `.venv`
- `dist`
- `build`
- `__pycache__`
- 自动生成文件
- 第三方 vendor 文件

### 3.3 300 行以上的处理要求

超过 300 行但未到 500 行时，不要求立刻拆分，但必须输出判断：

1. 文件当前承担了几个职责。
2. 是否已经有明显的业务边界可拆。
3. 拆分是否会影响当前功能。
4. 如果暂不拆，原因是什么。
5. 下次触发拆分的条件是什么。

### 3.4 500 行以上的处理要求

超过 500 行时，默认必须拆分。

允许豁免的情况：

- 文件是自动生成的。
- 文件是低风险静态配置或 schema 集合。
- 拆分会显著降低可读性，并且已有明确说明。

豁免要求：

- 必须写进 spec 或 review 结论。
- 必须说明为什么不能拆。
- 必须给出后续复查条件。

## 4. 文件级注释规则

所有文件级注释必须使用中文。允许保留必要的英文技术名词、类名、函数名、协议名和路径，但解释性文字必须是中文。

### 4.1 必须添加的位置

每个代码文件顶部必须有文件级说明：

- Python：模块 docstring。
- TypeScript / TSX：文件顶部块注释。
- 注释语言：中文。

### 4.2 注释必须讲清楚什么

每个文件级说明至少包含：

1. 这个文件负责什么。
2. 这个文件不负责什么。
3. 主要输入和输出是什么。
4. 依赖边界是什么。
5. 如果是适配层，要说明它适配谁。
6. 如果是业务编排层，要说明它调用哪些能力，但不直接实现哪些能力。

### 4.3 注释示例

Python 示例：

```python
"""应用层聊天编排模块。

本模块负责协调缓存查询、Agent 执行、LLM 流式生成和观测数据写入。
它不直接实现存储客户端、检索算法、HTTP 传输或前端展示格式。
"""
```

TypeScript / TSX 示例：

```ts
/**
 * 网关前端契约工具。
 *
 * 本文件集中维护后端接口路径和 SSE 帧解析逻辑。
 * UI 组件应该依赖解析后的帧，而不是直接依赖后端原始 payload 结构。
 */
```

### 4.4 禁止的注释

不接受以下类型的注释：

- 只重复文件名，比如“this is routes file”。
- 只描述语法，比如“定义函数”。
- 和实际职责不一致。
- 写得过长，变成设计文档。
- 用英文写解释性文字。

## 6. 实施阶段

### Phase A: 建立检查规则

目标：

- 新增脚本扫描源码行数。
- 输出超过 300 行和 500 行的文件列表。
- 默认排除第三方、构建产物和虚拟环境目录。

建议脚本：

- `tools/check_file_size.py`

验收：

- 可以输出 `warning >= 300`。
- 可以输出 `error >= 500`。
- CI 或本地验证可以直接运行。

### Phase B: 添加文件级注释

目标：

- 给所有业务源码文件补文件级说明。
- 优先处理 `src` 和 `frontend/src`。
- 不修改业务逻辑。
- 所有新增或修正的文件级说明必须使用中文。

验收：

- Python 文件顶部有模块 docstring。
- TS / TSX 文件顶部有块注释。
- 注释能说明职责边界。
- 注释解释性文字为中文。

### Phase C: 拆分 300 行以上文件

目标：

- 先拆 `orchestrator.py`。
- 再拆 `TracesPage.tsx`。
- 每次拆分后跑对应验证。

验收：

- 拆分后单文件低于 300 行。
- 当前功能不回退。
- 前端 `npm.cmd run build` 通过。
- 后端至少 `compileall src` 通过。

### Phase D: 固化为长期规则

目标：

- 把文件行数检查纳入日常验证。
- 后续新文件或改动文件如果超过阈值，必须提示。

验收：

- 每次 review 能看到文件行数风险。
- 500 行以上文件不能无说明合入。

## 7. 风险与控制

风险 1：为了降行数而过度拆分。

控制方式：

- 只按真实职责边界拆。
- 不拆出只有一两个函数的无意义文件。
- 拆分后调用链必须更清晰。

风险 2：注释变成形式主义。

控制方式：

- 注释必须讲职责边界，不写语法复述。
- 注释和文件实际内容不一致时，以代码为准并修注释。

风险 3：拆分影响当前功能。

控制方式：

- 每次只拆一个文件或一个职责组。
- 保持公共入口不变。
- 先迁移纯函数，再迁移流程函数。

## 8. 验收标准

这两项治理完成后，应满足：

1. `src` 和 `frontend/src` 中没有 500 行以上业务文件。
2. 300 行以上文件都有明确提醒和拆分判断。
3. 每个业务代码文件都有文件级职责说明。
4. 文件级职责说明使用中文。
5. `orchestrator.py` 和 `TracesPage.tsx` 已拆分或有你确认的暂缓理由。
6. 前端构建通过。
7. 后端语法编译通过。
8. 没有因为拆分改变现有接口、路由、SSE 协议或 UI 行为。
